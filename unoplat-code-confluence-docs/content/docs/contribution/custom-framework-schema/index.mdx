# Deterministic Interface Discovery Schema (DIDS)

The **Deterministic Interface Discovery Schema (DIDS)** enables contributors to define *framework and library patterns* for **Unoplat Code Confluence** to detect and analyze across different programming languages. This schema provides a **structured way** to describe how frameworks and libraries should be identified in codebases.

<Callout type="info" title="Quick Reference">
  - **Latest Schema**: [custom-framework-lib-schema.json](/schemas/custom-framework-lib-schema.json) *(opens as JSON)*
  - **Schema v4**: [custom-framework-lib-schema-v4.json](/schemas/custom-framework-lib-schema-v4.json) *(opens as JSON)*
  - **Schema v3 (legacy)**: [custom-framework-lib-schema-v3.json](/schemas/custom-framework-lib-schema-v3.json) *(opens as JSON)*
  - **Specification**: [JSON Schema Draft 2020-12](https://json-schema.org/specification)
  - **Organization**: By *programming language* → *library* → *capability* → *operation* → *detector*
</Callout>

<Callout type="info" title="Source of Truth">
  - Canonical definitions live in ingestion: [framework-definitions directory](https://github.com/unoplat/unoplat-code-confluence/tree/dev/unoplat-code-confluence-ingestion/code-confluence-flow-bridge/framework-definitions)
  - Docs consume generated static assets from: `/framework-definitions`
  - The published **v4 schema** is the next-generation contract for architecture-level grouping; the generated framework catalog remains v3-shaped until ingestion migrates
</Callout>

## Background and Problem Statement

### Background

Modern code intelligence systems usually cannot read entire repositories for every request. In large codebases, they typically rely on a retrieval pipeline:

1. Chunk source files into indexable units
2. Compute embeddings for semantic lookup
3. Retrieve nearest candidates from the index
4. Rerank candidates for relevance
5. Assemble final context for analysis or generation

This pattern works well for broad navigation and question answering, but it is fundamentally probabilistic. Retrieval quality depends on chunking boundaries, embedding behavior, reranking heuristics, and prompt phrasing.

At scale, this pipeline is also expensive to execute and maintain: indexing/embedding jobs must be recomputed, vector infrastructure must be operated, and retrieval quality must be continuously tuned. The approach is also non-extensible for deterministic interface discovery because framework-specific logic is usually implemented as ad-hoc runtime behavior instead of a reusable contract.

For interface discovery, we need stronger guarantees than "likely relevant context." We need reproducible detection of framework and library constructs so downstream systems can build stable blueprints, generate reliable structured output, and reason across repositories consistently.

The **Deterministic Interface Discovery Schema (DIDS)** introduces that missing contract: a versioned, machine-enforceable way to describe what to detect and how those detections should be interpreted.

### Problem Statement

For any coding task, an agent or human needs a reliable blueprint of the codebase. Today, that blueprint is inferred mostly from semantic retrieval and ad-hoc heuristics, which creates following problems:

1. **Non-determinism**: the same repository can produce different detections across runs.
2. **High execution and maintenance cost**: embeddings, indexes, and tuning loops create ongoing operational overhead.
3. **Low extensibility**: adding support for new frameworks or internal patterns requires custom, scattered logic.
4. **Weak verifiability**: behavior is hard to validate in CI because rules are implicit instead of contract-driven.
5. **Lack of open standards**: without an open, versioned schema, definitions are harder to share, interoperate, and evolve safely.

We therefore need a deterministic, schema-first standard that defines framework and library capabilities, operations, and detectors with explicit constraints and validation rules, enabling consistent interface discovery across any tech ecosystem.


## Goals

This specification work aims to:

1. Define a machine-enforceable detection contract for framework and library capabilities based on programming constructs like annotations, calls, inheritance, and imports.
2. Improve contributor correctness with explicit constraints and validation expectations
3. Support safe schema evolution with clear compatibility and migration expectations

## Capability, Operation, And Detector Hierarchy

Schema v4 separates **architectural meaning** from **precise detection identity**:

<Callout type="info" title="Three Layers Of Meaning">
  - **Capability** answers *what part of the system this belongs to* such as `http_endpoint`, `authentication`, or `file_storage`
  - **Operation** answers *what architectural action is happening* such as `register_handler`, `sign_in`, or `upload`
  - **Detector** answers *exactly which code pattern should match* using `absolute_paths`, `concept`, `target_level`, and optional `construct_query`
</Callout>

### Capability Definition Structure

Each **capability** groups related operations under a library-agnostic architectural family.

### `concept` Values

<TypeTable
  type={{
    AnnotationLike: {
      description: 'Decorators, annotations, and attributes that modify code behavior',
      type: 'enum',
    },
    CallExpression: {
      description: 'Function or method calls that create, configure, or invoke behavior',
      type: 'enum',
    },
    Inheritance: {
      description: 'Class inheritance patterns where a class extends a base class',
      type: 'enum',
    },
    FunctionDefinition: {
      description: 'Exported function declarations used as framework entrypoints',
      type: 'enum',
    },
  }}
/>

<Callout type="info" title="Concept Examples">
  - **AnnotationLike**: `@app.get("/")`, `@Component`, `[HttpGet]`, `@dataclass`
  - **CallExpression**: `mapped_column()`, `Field()`, `relationship()`, `useState()`
  - **Inheritance**: `class User(BaseModel)`, `class MyView(APIView)`, `class App extends Component`
  - **FunctionDefinition**: `export async function GET(request: Request) { ... }`
</Callout>

### Grouping Rules

<Callout type="warn" title="Group At The Right Layer">
  - Put **multiple operations** under one capability when they belong to the same architectural workflow, such as `authentication -> sign_in`, `sign_out`, and `observe_state`
  - Put **multiple detectors** under one operation when they all represent the same architectural action, such as `sign_in -> email_password`, `popup`, and `redirect`
  - Put **multiple absolute paths** inside one detector only when they share the same operation meaning, `target_level`, `concept`, `construct_query`, confidence semantics, and notes
  - If the variant changes the semantics that matter downstream, keep separate detectors instead of collapsing everything into one broad regex
</Callout>

### Detection Heuristics (v4)

Detection is still import-gated and regex-refined, but architectural grouping now lives above the precise detector layer. `absolute_paths` must be present in the file imports, and `construct_query` refines the tree-sitter patterns used to match decorators, calls, inheritance, and exported function declarations.

<Callout type="info" title="Low-Confidence Guidance">
  If a detector uses `base_confidence < 0.70`, include disambiguation instructions in `notes` so downstream validation can reliably confirm or reject ambiguous matches.
</Callout>

## Real Framework Examples

### FastAPI v4 Example

This example separates endpoint registration from router composition and keeps HTTP verbs as precise detectors under the same operation.

```json
{
  "python": {
    "fastapi": {
      "docs_url": "https://fastapi.tiangolo.com",
      "description": "Modern, fast web framework for building APIs with Python",
      "capabilities": {
        "http_endpoint": {
          "description": "Inbound HTTP route handlers",
          "operations": {
            "register_handler": {
              "description": "Register HTTP route handlers by verb",
              "startpoint": true,
              "detectors": {
                "get": {
                  "description": "GET route handler decorator",
                  "absolute_paths": [
                    "fastapi.FastAPI",
                    "fastapi.applications.FastAPI",
                    "fastapi.APIRouter",
                    "fastapi.routing.APIRouter"
                  ],
                  "target_level": "function",
                  "concept": "AnnotationLike",
                  "construct_query": {
                    "method_regex": "get"
                  }
                },
                "post": {
                  "description": "POST route handler decorator",
                  "absolute_paths": [
                    "fastapi.FastAPI",
                    "fastapi.applications.FastAPI",
                    "fastapi.APIRouter",
                    "fastapi.routing.APIRouter"
                  ],
                  "target_level": "function",
                  "concept": "AnnotationLike",
                  "construct_query": {
                    "method_regex": "post"
                  }
                }
              }
            }
          }
        },
        "router_composition": {
          "description": "Router organization and inclusion",
          "operations": {
            "compose_router": {
              "description": "Include routers into a larger route tree",
              "detectors": {
                "include_router": {
                  "absolute_paths": [
                    "fastapi.FastAPI",
                    "fastapi.applications.FastAPI",
                    "fastapi.APIRouter",
                    "fastapi.routing.APIRouter"
                  ],
                  "target_level": "function",
                  "concept": "AnnotationLike",
                  "construct_query": {
                    "method_regex": "include_router"
                  }
                }
              }
            }
          }
        }
      }
    }
  }
}
```

## Validation Rules

<Callout type="warn" title="v4 Mandatory Constraints">
  The schema enforces these **mandatory constraints**:
  - Capability keys must match the published capability enum
  - Operation keys must match the published operation enum
  - **CallExpression** detectors *must* use `target_level: "function"` and define `base_confidence`
  - Low-confidence **CallExpression** detectors with `base_confidence < 0.70` *must* include non-empty `notes`
  - **Inheritance** detectors *must* use `target_level: "class"`
  - **FunctionDefinition** detectors *must* use `target_level: "function"`

  Violations will cause schema validation to fail.
</Callout>

## Best Practices

### 1. Comprehensive Coverage
- Define **all major capabilities** of a framework or library first
- Group related architectural actions under **operations** before adding precise detectors
- Cover both *common* and *advanced* detector variants when they are semantically important

### 2. Accurate Paths
- Use **fully-qualified import paths**
- Test paths against actual library documentation
- Include `version` information when patterns change between versions

### 3. Clear Descriptions
- Write **human-readable descriptions**
- Explain the *purpose* and *context* of each capability and operation
- Include examples in `notes` when helpful

### 4. Group At The Right Layer
- Put workflow families such as authentication or HTTP routing at the **capability** layer
- Put actions such as `sign_in`, `register_handler`, or `upload` at the **operation** layer
- Put exact SDK or framework variants at the **detector** layer
- Use `startpoint: true` on operations that represent **entry points** like API route registration or message consumption

### 5. Strategic Regex And Confidence Usage
- Use `construct_query` regex patterns to **refine detector matches**
- Test regex patterns against real code examples
- Keep patterns *specific enough* to avoid false positives
- Use lower `base_confidence` only when ambiguity is real, and document disambiguation in `notes`

## Contributing Schema Updates

To contribute new framework/library definitions:

1. **Fork the repository** and create a feature branch
2. **Update the versioned schema file** at `/public/schemas/custom-framework-lib-schema-v4.json`
3. **Keep the latest alias** at `/public/schemas/custom-framework-lib-schema.json` aligned with the newest published version
4. **Add comprehensive capability, operation, and detector definitions** following the structure above
5. **Test your definitions** against real codebases
6. **Submit a pull request** with detailed examples and test cases

### Testing Your Schema

Before submitting, verify your schema definitions:

1. **JSON Validation** — Ensure the file is valid JSON
2. **Schema Validation** — Validate against the [JSON Schema specification](https://json-schema.org/specification)
3. **Architecture Review** — Confirm the capability and operation choices reflect software-engineering fundamentals instead of raw API names
4. **Real-world Testing** — Test detectors against actual codebases using the framework
5. **Documentation** — Ensure all referenced URLs are accessible

## Troubleshooting

### Common Issues

<Accordions>
  <Accordion title="Invalid capability key" id="error-capability-key">
    **Error Message:**
    ```
    Error: property name must be equal to one of the allowed values
    ```

    **Cause:** A capability name such as `auth_flow` or `firebase_auth` is not part of the published capability enum.

    **Solution:** Use a library-agnostic capability family such as:
    - **Correct**: `authentication`, `http_endpoint`, `file_storage`
    - **Incorrect**: `get_auth`, `firebase_auth`, `route_get`
  </Accordion>

  <Accordion title="Missing detectors or required detector fields" id="error-required">
    **Error Message:**
    ```
    Error: Missing required property 'detectors'
    ```

    **Cause:** An operation does not define detectors, or a detector is missing one or more required fields.

    **Solution:** Include **all required fields** at the correct layer:
    - Capability: `description`, `operations`
    - Operation: `description`, `detectors`
    - Detector: `absolute_paths`, `target_level`, `concept`

  </Accordion>

  <Accordion title="Concept-Target Level Mismatch" id="error-mismatch">
    **Error Message:**
    ```
    Error: CallExpression requires target_level 'function'
    ```

    **Cause:** The detector's `concept` and `target_level` values are incompatible.

    **Solution:** Follow the detector-level concept-target level constraints:
    - `CallExpression` → `target_level: "function"`
    - `Inheritance` → `target_level: "class"`
    - `AnnotationLike` → can be either `"function"` or `"class"`
  </Accordion>

  <Accordion title="Detector Not Detected" id="error-not-detected">
    **Symptom:** Your schema is valid but the detector isn't being detected in code.

    **Possible Causes:**
    1. **Incorrect `absolute_paths`** — Verify the exact import path used in the library
    2. **Over-grouped detector** — Split variants into separate detectors when verbs, providers, or flows need different semantics
    3. **Missing `construct_query`** — Add regex patterns to narrow down matches
    4. **Import not present** — Detection is import-gated; ensure the file imports the library

    **Solution:** Test with a minimal code example and check the Tree-sitter output.
  </Accordion>
</Accordions>

## Future Enhancements

The schema is designed to be **extensible**. Future versions may include:
- *Additional programming languages* (Rust, C#, Kotlin)
- *Capability-specific operation profiles* where certain capability families can narrow the allowed operation set further
- *Richer qualifier metadata* for cases like HTTP verbs, auth providers, and transport protocols
- *Migration tooling* to convert flat v3 `features` into v4 `capabilities -> operations -> detectors`
- *Ingestion and query-engine adoption* of the v4 grouping model across the full stack
